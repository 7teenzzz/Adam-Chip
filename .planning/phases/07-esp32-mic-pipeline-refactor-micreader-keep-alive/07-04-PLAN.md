SSE events from Orchestrator that drive chat.js / wakeMeter.js:

- `voice_state_change` (emitted by VoiceLoopController._set_voice_state) — payload: `{from: str, to: str, reason: str}`. New states in scope: `boot_warmup`, `standby`, `listening`, `reply`.
- `voice_loop_started` — emitted by VoiceLoopController.start() AFTER `_set_voice_state("boot_warmup", "voice_loop_started")`. Payload is the full `status()` dict; `payload.voice_state === "boot_warmup"`.
- `voice_loop_stopped` — payload is the full `status()` dict with `voice_state === "standby"`.
- `audio_level` — emitted by MicReader. Payload: `{level: number, state: string, source: string, channels?: 1|2, level_l?: number, level_r?: number}`. The `state` field now includes `"boot_warmup"` for the duration of warmup TTS (because MicReader emits while voice_loop._voice_state is still boot_warmup). `source` value `"esp32_stereo"|"esp32_mono"|"connecting"|"failed"`.

Initial snapshot from `/api/agent/status`:
- `voice_loop.voice_state` may now be `"boot_warmup"` on a freshly-started Orchestrator. Existing code reads `vl.running` to derive label — that path must also recognize boot_warmup.

Existing constants in chat.js to extend:
- `HEARING_COLORS` (lines ~189-198) — add `boot_warmup` key.
- `HEARING_LABELS` (lines ~199-208) — add `boot_warmup: "⌛ Инициализация"` (this label currently lives on `loading`, but the semantics differ: `loading` = voice_loop down; `boot_warmup` = up but not yet listening).
