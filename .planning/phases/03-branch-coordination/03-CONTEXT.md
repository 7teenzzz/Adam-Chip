# Phase 3 Context: Branch Coordination

## Team

2 разработчика × 2 Claude-аккаунта каждый = до 4 агентов, работающих в одном репозитории.

## Problem Statement

Агент, переключившийся на незнакомую ветку (`V-S06.3`, `ESP32-sound-out`, etc.), не получает никакого контекста:
- Зачем эта ветка существует
- Кто/что на ней работает прямо сейчас
- Какие файлы затрагиваются (риск конфликта с другой веткой)
- Когда и при каких условиях она мёржится
- Является ли изменение глобальным (затронет main) или локальным экспериментом

Параллельно — нет одного места, где видно «что сейчас в полёте» по всем активным веткам.

## Decisions

**Два слоя:**

**Слой 1 — `BRANCH.md` (per-branch, в корне репозитория)**
- Создаётся при ветвлении от main
- Удаляется командой `git rm BRANCH.md` при мёрже — без архивирования
- Содержит: цель ветки, статус, modified areas, условие мёржа, global changes flag, notes for agents
- Идентификатор = имя ветки (нет поля Owner, нет личных имён)
- Для агента: первый файл для чтения при переключении на не-main ветку

**Слой 2 — `.planning/ACTIVE.md` (глобальный, в main)**
- Таблица: ветка / статус / modified areas / merge blocker
- Обновляется только в двух точках: создание ветки и её закрытие/мёрж
- Нет колонки Owner — ветка сама является идентификатором
- Merge-конфликт минимален: каждая ветка редактирует только свою строку

## Scope (что входит)

1. `docs/BRANCH-template.md` — шаблон BRANCH.md + конвенция в одном файле
2. `.planning/ACTIVE.md` — таблица активных веток (заполнить текущее состояние)
3. Обновить `CLAUDE.md` — Reading Order: «если не на main — прочитай BRANCH.md первым»
4. Обновить `STATE.md` — ссылка на ACTIVE.md

## Scope (что НЕ входит)

- Ретроактивное создание BRANCH.md для существующих веток — только шаблон + конвенция
- Скрипты автоматизации (создание BRANCH.md при `git checkout -b`) — можно позже
- Архивирование BRANCH.md — явное решение: удалять без следа

## ACTIVE.md status values

| Статус | Значение |
| --- | --- |
| `experimenting` | Активный эксперимент, результат неизвестен |
| `ready-for-review` | Готова к мёржу, ждёт ревью |
| `blocked` | Заблокирована: HW, зависимость, тест |
| `stale` | Не обновлялась > 2 недель, судьба неизвестна |

## Global changes field

Ключевое поле в BRANCH.md. Если ветка изменяет Config.json, API, схему данных или архитектурное решение — это **global change**: нужна координация перед мёржем. Если поле `нет` — чистый эксперимент, мёрж не требует согласования.
