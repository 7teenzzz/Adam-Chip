**User's choice:** «в момент когда оркестратор завис ничего не воспроизводилось (адам молчал). я предполагаю, что проблема связана с реализацией режима reply. правильная логика reply должна быть идентична логике listening, кроме временного ограничения (через 4 секунды после перехода в режим reply, если пользователь не сделал следующий запрос, то переходим в режим standby). проверь что reply соответствует ожидаемой логике. дополнительно проверь reply на избыточность»
**Notes:** Критическое уточнение — TTS не воспроизводился в момент hang → эхо физически невозможно → root cause не в акустике, а в реализации reply mode. Сменили направление обсуждения с «как подавить эхо» на «как упростить reply mode».

## Подход к фиксу reply mode

| Option | Description | Selected |
|--------|-------------|----------|
| Полный рефактор (рек) | Reply = listening + один таймер. Убираем _REPLY_GUARD_SEC, дублирование, absolute_deadline | частично ✓ |
| Точечный фикс | Оставить структуру, починить только конкретный hang (debounce + логи) | |
| Диагностика первой | Сначала воспроизвести hang + SIGUSR1, потом выбирать фикс | |

**User's choice (модификация рек-опции):** «полный рефактор. но нужно сохранить _REPLY_GUARD_SEC=0.6. убрать дублирование accumulation/endpointing. двойной таймаут - правильная логика (если 4 секунд молчания то переходим в standby, второй таймаут должен работать также как в listening: если пользователь начал говорить запрос, есть максимально возможное окно для запроса - защита от бесконечной диктовки). по итогу: логика reply должна быть идентичной listening + если 4 секунды молчания - переход в standby»
**Notes:** Уточнения к рекомендованной опции:
- `_REPLY_GUARD_SEC=0.6` сохранить (защитный буфер).
- «двойной таймаут» переосмыслен: один таймер silence (4 сек) + max_segment_ms (тот же что listening) для защиты от бесконечной диктовки. Отдельный `reply_absolute_deadline_sec` не нужен — его роль теперь играет общий `max_segment_ms`.

## Стратегия расследования 6-минутного hang

| Option | Description | Selected |
|--------|-------------|----------|
| Рефактор сначала | Рефактор reply, если hang повторится — отдельная фаза | |
| Рефактор + логи переходов voice_state | Рефактор + state_changed/heartbeat события | ✓ |
| Рефактор + SIGUSR1 dump | Максимум диагностики сразу, но больше scope | |

**User's choice (после уточняющего диалога):** «Да, это скоп» (на формулировку «рефактор reply + детальные логи переходов voice_state для диагностики hang»)
**Notes:** Пользователь сначала переспросил «что такое 6-минутный hang» — пришлось объяснить детали из ROADMAP Symptoms (события в events.jsonl пропадают на 6 минут, UI VU замерзает, Адам не реагирует на wake word). Затем уточнил: «в 21:04:15 оркестратор завис, а спустя 6 минут я выключил адама» — то есть hang перманентный (до ручного выключения). Это укрепило аргумент за обязательность диагностических логов в этой фазе (SIGUSR1 — в отложенную фазу).
