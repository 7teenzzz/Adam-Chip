## I. Инвентаризация проблем

### 1. Отсутствие Section 3.0 (bridging section)

**Проблема:** Глава 3 начинается с 3.1 без явной связи с теорией (главы 1–2). Читатель не видит, как 8 критериев из 2.1 переводятся в инженерные требования.

**Ошибка структуры:** Читатель доходит до конца 3.4 и всё ещё не понимает: какой критерий какая метрика покрывает?

**Решение:** 
- Создать Section 3.0 (~800 слов)
- Явно мэпить: 2.1.1–2.1.8 ↔ компоненты 3.X ↔ метрики 3.4.2–3.4.4
- Таблица соответствия (см. ниже: **Mapping Table**)

### 2. AIIM Framework рассыпан по тексту

**Проблема:** "AIIM" упоминается в:
- 3.1.1 «Формула системного промпта» (orphan, стр 75)
- 3.2.3 «Системный промпт и конфигурация идентичности» (стр 89)
- «Agent Adam Chip/About/Identity.md» (в persona_paths)
- Нет единого определения

**Ошибка:** Читатель не понимает: AIIM это что? Формула? Framework? Как она связана с Config.json?

**Решение:**
- Выбрать ONE canonical location (рекомендую 3.2.3 как Methods для идентичности)
- Создать таблицу: AIIM-компоненты ↔ Config.json параметры
- Убрать повтор из 3.1.1 и перенести в 3.2.3 с ссылкой
- В 3.1.1 оставить только концептуальную основу (зачем нужна идентичность)

### 3. Проактивное поведение рассеяно (crit-8: emergence)

**Проблема:** Упоминается в:
- 3.1.2 «Логика поведения агента» (описание)
- 3.3.2 «Перцептивный и моторный слои» (примеры с PWM)
- 3.3.4 «Сценарий взаимодействия» (примеры диалогов)
- Без единого определения

**Ошибка:** Критерий 2.1.8 (уровень эмерджентности) не явно покрывается ни одной метрикой в 3.4.

**Решение:**
- В 3.1.2 дать определение: "проактивное поведение = инициатива без явного запроса (интерпретация сцены → предложение)"
- В 3.2.1 показать архитектурно: где в pipeline это реализовано (inference → scene + mood scoring → initiative flag)
- В 3.3.4 привести примеры: когда агент инициировал (сценарии из событий)
- В 3.4.4 добавить метрику: RI (Rate of Initiative) — процент turn-ов, где агент первый предложил действие

### 4. Метрики не мэппены на критерии 2.1

**Проблема:** 3.4.2–3.4.4 определяют метрики (RAS, RDI, NVR, RI, CRS, LMRR, SCS, SIAR, latency), но не связывают их с criteria 2.1.1–2.1.8.

- 3.4.3 (NVR, RI, CRS) → presumably 2.1.5 (temporal coherence)

- Ничего явно не покрывает 2.1.1 (autonomization), 2.1.2 (agency type), 2.1.7 (embodiment)

**Ошибка:** Читатель не может сказать: "результаты показывают, что критерий 2.1.X удовлетворён на Y%"

**Решение:** Создать таблицу (в Section 3.0 и переиспользовать в 3.4.5):

| 2.1.1 | Степень автономизации | Action layer | RI, response latency | …% инициативы |
| 2.1.2 | Тип агентности | Config: tuning.json → prompt | SIAR (scene awareness) | … |
| 2.1.3 | Устойчивость идентичности | AIIM + System.md | RAS (role adherence) | … |
| 2.1.4 | Режим нормативности | EchoGate + tuning | RDI (drift index) | … |

### 5. Перекрытие содержания между секциями

**Проблема:**
- 3.1.2 (логика поведения) описывает инициативу → повторяется в 3.3.4 (сценарии)
- 3.2.1 (общая архитектура) описывает компоненты → повторяется в 3.2.2–3.2.6 (детали)
- 3.2.5 (перцептивные контуры) и 3.3.2 (перцептивный слой) почти дублируют друг друга

**Ошибка:** Читатель тратит время на чтение одного и того же дважды. Глава раздувается без добавления информации.

**Решение:**
- 3.1.2 → только концепция и типология поведения (2–3 параграфа)
- 3.3.4 → примеры реальных сценариев из логов (конкретные случаи, когда агент инициировал)
- 3.2.1 → диаграмма + описание архитектуры
- 3.2.2–3.2.6 → детали компонентов БЕЗ повтора архитектуры
- 3.2.5 и 3.3.2 → разделить так: 3.2.5 (приложение) ↔ 3.3.2 (инсталляция), один не повторяет другого

## II. Root Cause Analysis

**Основная причина:** Глава 3 писалась как последовательное описание компонентов (что существует), а не как систематическая верификация теории (что доказано).

**Следствия:**
1. Нет явной связи с критериями 2.1 → читатель теряется
2. Дублирование → глава раздувается без смысла
3. Примеры спорадичны → непонятно: правда ли работает?
4. Метрики определены, но не интерпретированы → так и не ясно: квазисубъектность достигнута?

## V. IMRAD Template для каждой секции

Используй этот шаблон при переписании каждой подсекции:

```markdown
### Раздел X.Y.Z: [Название]

#### Introduction (Why?)
- Почему этот компонент критичен для критерия 2.1.N?
- Какие теоретические основания (из 1.X и 2.X)?
- Если это part of larger section, как связано с другими?
- Обоснование из конструктивных требований (3.0 mapping table)

#### Methods (How?)
- Архитектура: какие модули, как они связаны?
- Параметры из Config.json / Config.schema.json
- Алгоритм или формула (если есть)
- Примеры кода ИЛИ выжимка из конфига
- Версии, paths, dependencies

#### Results (What?)
- Примеры работы (если это application component) ИЛИ примеры результатов (если это metrics section)
- Таблица результатов / примеры диалогов / метрики
- Логов и данные (если доступны)
- Графики (если есть временные ряды)

#### Analysis/Discussion (So what?)
- Как это реализует конкретный критерий 2.1.N?
- Какой уровень достигнут: FULL / PARTIAL / MISSING?
- Ограничения и trade-offs
- Сравнение с case-studies из 2.2–2.3 (если релевантно)

**Word count:** [target, e.g. 800 слов]
```
