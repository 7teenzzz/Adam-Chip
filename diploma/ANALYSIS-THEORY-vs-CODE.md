### 3.2.6 Командный контур и связь с исполнительной частью: **CRITICAL GAP ⚠️**

**Диплом (стр. 98):**
> Commander.py выделяет из общего ответа краткие маркеры состояния, заранее ограниченные допустимым набором: например, `[радость]`, `[удивление]`, `[грусть]`, `[настороженность]`.

**Код (System/adam/action.py):**

```python
Mood = Literal["neutral", "curious", "warm", "irritated", "silent"]

class Action:
    kind: ActionKind  # "no_action" | "scene" | "channel"
    mood: Mood
    scene: str | None  # e.g. "boot_idle", "alternating"
    channel: int | None
    value: int | None
    duration_ms: int

def infer(self, reply_text: str, context: dict) -> Action:
    # Простой парсинг: ищет ключевые слова в reply_text
    if any(word in text for word in ("рад", "интересно", "вижу", "привет")):
        return Action(kind="scene", mood="warm", scene="alternating", ...)
    if any(word in text for word in ("нет", "осторож", "не могу")):
        return Action(kind="scene", mood="irritated", scene="boot_idle", ...)
```

**Проблема:** ⚠️ **МИСМАТЧ**

| Аспект | Диплом | Код |
|--------|--------|-----|
| **Парсинг маркеров** | LLM выделяет чёткие теги типа `[радость]` | Простое keyword matching на основе русских слов |
| **Набор эмоций** | 4+ эмоции (радость, удивление, грусть, настороженность) | 5 mood: neutral, curious, warm, irritated, silent |
| **Исполнительные команды** | Музыкальная нотация (свечение, звук, вибрация в отдельных каналах) | Предзаданные сцены (boot_idle, alternating) или прямые PWM channel/value |
| **Сложность** | Архитектурно разделены: LLM → Commander → MCU | Всё в одном слое: Action.infer() с простыми rules |

**Реальное использование:**
- Action layer используется **после** синтеза TTS, не для парсинга команд из LLM-ответа
- Mood выбирается на основе ключевых слов в **LLM response**, не на основе явных маркеров
- Сцена (`scene`) или PWM (`channel`/`value`) отправляются на ESP32

**Вывод:** ✗ **CRITICAL GAP** — Диплом описывает более сложный и архитектурно чистый командный контур (Commander.py с явными тегами), чем что реально реализовано. Код использует более упрощённый подход на основе keyword matching.

## Критичные гэпы (для доработки)

### Гэп 1: Командный контур (3.2.6) — **HIGH PRIORITY**

**Проблема:** Диплом описывает архитектурно чистый подход с явными эмоциональными тегами, которые LLM должна генерировать. Код упрощает это до keyword matching.

**Вариант исправления:**

Вариант A (High-effort):
- Модифицировать LLM prompt, чтобы LLM генерировала явный тег типа `[mood: warm]` или `[scene: alternating]`
- Парсить эти теги в action.py
- Это требует переучивания поведения LLM через tuning.json

Вариант B (Low-effort):
- Оставить keyword matching, но документировать это как "упрощение для производства"
- Добавить comment в code, указывающий на диплом как на идеальный дизайн

**Рекомендация:** Вариант B для текущей фазы. Вариант A — в следующей фазе, когда будет time для tuning.

### Гэп 2: Метрики тестирования (3.4) — **MEDIUM PRIORITY**

**Проблема:** Диплом описывает метрики (RAS, RDI, CRS, LMRR), которые требуют экспертной разметки. Код предоставляет логи, но не сами метрики.

**Вариант исправления:**

Вариант A: Создать скрипт для экспортирования turn-ов в формат для экспертной разметки
- Экспорт из `/api/agent/turns` в CSV/JSON для ручной разметки
- Затем загрузить разметку и рассчитать метрики

Вариант B: Реализовать автоматические приблизители метрик (без экспертизы)
- RAS: LLM-based classifier (use gemma или claude для оценки соответствия роли)
- RDI: дрейф ролевых паттернов через embedding similarity
- NVR: rule-based checker на основе нормативных constraint-ов
- RI: word frequency анализ

**Рекомендация:** Вариант A (экспорт для разметки) для Phase 7. Вариант B (автоклассификация) — как опциональное расширение.
