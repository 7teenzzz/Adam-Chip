# 3.2.1 Общая архитектура нейроагентного приложения

В данном разделе описывается общая архитектура программного обеспечения нейроагентной системы «Адам Чип»: принципы организации, состав функциональных блоков и роль центрального диспетчера. Архитектурные решения направлены на обеспечение модульности, трассируемости событий и возможности горячей перезагрузки конфигурации.

## Архитектурные принципы

Система построена на основе **событийно-ориентированной модульной архитектуры**. Центральный принцип: каждый функциональный блок изолирован, взаимодействует с остальными исключительно через общий event bus (`System/adam/events.py`) и HTTP-интерфейсы. Конфигурация системы централизована в `System/Config.json` и `System/Config.schema.json`; числовые параметры не хардкодируются в логику модулей.

Главная точка входа — `System/Orchestrator.py`: файл FastAPI-приложения, запускающий asyncio event loop и координирующий все модули. В граф-отчёте кодовой базы `Orchestrator.py` является god-node с 85 рёбрами — наибольший hub в графе. Такое центральное положение отражает роль оркестратора: он не выполняет бизнес-логику самостоятельно, но управляет жизненным циклом, маршрутизирует события и координирует фоновые задачи.

## Функциональные блоки

Архитектура разделена на пять функциональных блоков, соответствующих пяти функциям нейроагента (раздел 3.1.3):

**Блок Speech (Речь)** включает ASR-клиент (`WhisperASRClient` в `System/adam/inference.py`) и TTS-клиент (`TTSClient` в `System/adam/inference.py`), а также детектор голосовой активности (`System/adam/webrtc_vad.py`) и систему ключевого слова (`System/adam/wake_word.py`). Блок управляет полным аудиопipelinom: захват микрофона → VAD → ASR → TTS → ALSA-воспроизведение.

**Блок Interlayers (Промежуточный слой)** включает построитель промпта (`PromptBuilder` в `System/adam/prompt.py`) и слой действий (`ActionLayer` в `System/adam/action.py`). Блок преобразует сырой ввод в структурированный запрос к языковой модели и извлекает командные теги из ответа.

**Блок Memory (Память)** реализует многоуровневую систему хранения: `EpisodicMemory` и `MemoryStore` в `System/adam/memory.py`, `SessionAccumulator` в `System/adam/episodic.py`, механизм anti-repeat `EchoGate` в `System/adam/echoes_gate.py`.

**Блок Tools (Инструменты)** управляет взаимодействием с физическим слоем: `MCUClient` в `System/adam/device.py` (HTTP к ESP32), `CameraReader` для захвата видео, `SceneWorker` как фоновый VLM-анализатор.

**Блок Infrastructure (Инфраструктура)** обеспечивает наблюдаемость и управляемость: `EventLog` в `System/adam/events.py`, `TuningStore` в `System/adam/tuning.py` (горячая перезагрузка параметров), `MetricsCollector` в `System/adam/metrics.py`, `LogViewer` в `System/adam/log_viewer.py` (всегда доступный сервис на порту 8083).

## Диалоговый цикл

Стандартный диалоговый turn включает следующие стадии: PERCEPTION (захват аудио + VAD + wake word) → ORCHESTRATOR (диспетчеризация события) → CONTEXT (сборка промпта: PromptBuilder) → LLM (вызов языковой модели) → POSTPROCESS (ActionLayer: извлечение тегов) → EXECUTION (параллельно: TTS воспроизведение + MCU команды) → MEMORY (запись в SessionAccumulator и EventLog) → STANDBY (ожидание следующего ключевого слова). Каждый turn получает уникальный `turn_id` (короткий UUID), по которому события трассируются через `/api/agent/turns`.

## Операторский интерфейс

Для мониторинга и управления системой разработан web-интерфейс на основе FastAPI и WebUI (`System/adam/ui.py`, `System/HostUI/`, `System/WebUI/`). Интерфейс доступен по основному порту 8080 оркестратора и включает дашборд агента, отладочную страницу и панель логов. Отдельный сервис Log Viewer (`System/adam/log_viewer.py`) работает на порту 8083 независимо от оркестратора и сохраняет доступность даже при его падении — критически важное свойство для выставочного контекста.
