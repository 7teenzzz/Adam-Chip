# 3.3.5 Тестирование и апробация инсталляции

В данном разделе описывается методика тестирования инсталляции «Адам Чип»: набор автоматизированных диагностических скриптов, production runbook и процедуры проверки перед выставочным развёртыванием. Тестирование охватывает все ключевые подсистемы — от Jetson inference stack до ESP32 firmware и аудиотракта.

## Стратегия тестирования

Тестирование инсталляции строится на принципе **слоистой диагностики**: каждая подсистема проверяется независимо, после чего проводится интеграционная проверка взаимодействия. Автоматизированные скрипты обеспечивают воспроизводимость результатов и снижают зависимость от ручного контроля в условиях выставки.

Весь набор инструментов разработан на bash/Python без внешних зависимостей и запускается непосредственно на Jetson. Логи сохраняются в стандартный вывод и `data/adam/events.jsonl`.

## Набор диагностических скриптов

**`scripts/adam_healthcheck.sh`** — комплексная проверка работоспособности системы: доступность всех сервисов (LLM, TTS, ASR, VLM, Orchestrator, Log Viewer), ответ API-эндпоинтов, соответствие power mode требованиям MAXN. Запускается первым перед любым выставочным развёртыванием.

**`scripts/adam_media_probe.sh`** — диагностика медиапотоков: проверка ALSA-устройств (входных и выходных), доступности MJPEG-потока с ESP32, корректности устройств TTS-воспроизведения (`plughw:1,3`) и записи (`pulse`). Выявляет конфигурационные проблемы аудиотракта до старта оркестратора.

**`scripts/adam_torch_doctor.sh`** — диагностика PyTorch / CUDA / Silero на Jetson: проверка корректности установки NVIDIA-совместимого PyTorch, наличия CUDA-устройства, загрузки модели Silero. Критично для верификации после каждого обновления зависимостей.

**`scripts/adam_asr_cuda_check.sh`** — специализированная проверка CUDA-совместимости WhisperX: верификация ctranslate2 CUDA backend, доступности Whisper-модели, работоспособности ASR-сервиса в Docker. Выявляет типичную проблему Jetson: `pip install whisperx` устанавливает CPU-only ctranslate2.

**`scripts/adam_tts_smoke.sh`** — smoke-тест синтеза речи: генерация тестовой фразы через Silero TTS и воспроизведение через ALSA, верификация что звук действительно выводится на `plughw:1,3` (HDMI) без ошибок.

**`scripts/adam_service_status.sh`** — проверка статуса всех systemd-сервисов (`adam-llm`, `adam-tts-silero`, `adam-asr-whisperx`, `adam-orchestrator`, `adam-logviewer`).

**`scripts/adam_pull_logs.py`** — удалённый pull логов с Jetson для анализа на Windows/macOS. Поддерживает группировку по `turn_id`, фильтрацию по стадии (asr/llm/tts/action), live tail режим (`--follow`).

## Production Runbook

Процедура выставочного развёртывания задокументирована в `docs/RUNBOOK_JETSON_EXHIBITION.md`. Runbook описывает последовательность действий: подготовка окружения → запуск сервисов в правильном порядке → healthcheck → переключение в exhibition mode → верификация. Порядок запуска сервисов критичен: `adam-logviewer` запускается первым (всегда доступен), затем `adam-llm` (загрузка модели ~2 мин), TTS, ASR, наконец `adam-orchestrator`.

Переключение между режимами:
```bash
./scripts/adam_set_mode.sh exhibition   # включить wake word gate + power check
./scripts/adam_set_mode.sh maintenance  # отключить gate, для разработки
```

## Интеграционный тест: тестовый turn

Верификация полного диалогового цикла проводится через HTTP API:
```bash
curl --noproxy '*' -fsS http://127.0.0.1:8080/api/agent/turn \
  -H 'Content-Type: application/json' \
  -d '{"transcript":"Адам, ты меня слышишь?"}' | python3 -m json.tool
```

Результат проверяется по трём критериям: наличие LLM-ответа в поле `reply`, ненулевая латентность в `metrics`, корректный `action_result` (даже `no_action` является валидным результатом).
