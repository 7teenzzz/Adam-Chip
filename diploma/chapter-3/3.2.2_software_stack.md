# 3.2.2 Программный стек

В данном разделе описывается технологический стек нейроагентной системы: аппаратная платформа, операционная среда, компоненты вывода (inference) и пользовательский интерфейс. Отдельно фиксируются расхождения между ранними проектными решениями и реализованной конфигурацией, которые стали результатом инженерных компромиссов в ходе разработки.

## Аппаратная платформа

Вычислительным ядром инсталляции является NVIDIA Jetson Orin NX Super 16 GB — системный модуль на базе ARM-архитектуры (8 ядер ARMv8) с GPU Tegra Orin и унифицированной памятью 16 ГБ. Выбор платформы обусловлен сочетанием компактного форм-фактора, энергоэффективности и достаточной вычислительной мощности для параллельного выполнения языковой модели, синтеза речи, распознавания речи и VLM-анализа.

Для работы в выставочном режиме система переводится в режим максимальной производительности MAXN (`nvpmodel -m 0`) с принудительным выставлением частот (`jetson_clocks`). Соответствие этим требованиям контролируется модулем `power.py` при каждом старте оркестратора; при несоответствии старт в выставочном режиме блокируется.

Операционная система — Ubuntu 22.04.5 LTS с JetPack (пакет NVIDIA для Jetson, включающий CUDA, cuDNN, TensorRT).

## Контейнеризация и сервисная архитектура

Инференс-сервисы развёрнуты в Docker-контейнерах; процессная изоляция обеспечивается через systemd. Каждый сервис запускается как независимый юнит:

| Сервис | Runtime | Модель | Порт |
|--------|---------|--------|------|
| LLM | llama.cpp (OpenAI API) | gemma-4-E4B-it-UD-Q4_K_XL | 8081 |
| VLM | nano_llm (Docker) | VILA 1.5-3b | 8084 |
| ASR | WhisperX (CUDA, Docker) | small (ru-RU) | 8095 |
| TTS | Silero v5_5_ru | eugene | 8082 |
| Orchestrator | FastAPI + asyncio | — | 8080 |
| Log Viewer | FastAPI (always-on) | — | 8083 |

Такая архитектура позволяет независимо перезапускать сервисы, обновлять модели и проводить диагностику без остановки всей системы.

## Языковая модель: замена модели в ходе разработки

На этапе проектирования в качестве языковой модели предполагалось использование Cosmos Reasoning 2 2B. В процессе разработки модель была заменена на `gemma-4-E4B-it-UD-Q4_K_XL` (квантизация Q4_K_XL, формат GGUF). Причина замены: доступность данной квантизации для архитектуры aarch64 и практически достигнутые показатели скорости генерации ~65 токен/с на Jetson Orin NX, соответствующие требованиям латентности в выставочном контексте.

Важная особенность Gemma 4 E4B: модель использует **гибридную схему внимания** (global + sliding window 512), что обуславливает полный сброс KV-кэша между turn'ами и необходимость полного prefill (~2781 токен) при каждом обращении. Следствием является фиксированная задержка ~9 секунд на turn, компенсируемая механизмом фраз-заполнителей (см. раздел 3.2.5).

Дополнительно: модель функционирует в режиме **thinking model** и по умолчанию помещает генерацию в поле `reasoning_content`. Для корректного получения текстового ответа в production-сервисе llama.cpp используется флаг `--reasoning off`, прописанный в `deploy/systemd/adam-llm.service`.

## Оркестрация: Python + FastAPI + asyncio

Оркестратор написан на Python 3 с использованием asyncio для параллельной обработки событий. FastAPI обеспечивает REST API (порт 8080), SSE-поток событий и web UI. Выбор Python обусловлен наличием Jetson-совместимых колёс PyTorch, богатой экосистемой AI-библиотек и необходимостью интеграции с llama.cpp через OpenAI-совместимый API.

Особенности установки зависимостей на Jetson: Silero TTS и WhisperX устанавливаются с флагом `--no-deps` после Jetson-совместимого PyTorch — стандартный pip-резолвер заменяет NVIDIA-специфичный PyTorch на CPU-only версию, нарушая CUDA-совместимость.

## Интерфейс оператора: Web UI

Для мониторинга и управления системой в ходе выставки разработан web-интерфейс на базе FastAPI-бэкенда (`System/adam/ui.py`): дашборд агента, отладочная страница и просмотр логов. Доступ осуществляется через браузер по адресу `http://<IP-Jetson>:8080`. Отдельный сервис Log Viewer (`System/adam/log_viewer.py`) работает на порту 8083 как самостоятельный всегда-доступный процесс: он сохраняет работоспособность даже при падении оркестратора, что критически важно для диагностики в условиях публичного показа.
