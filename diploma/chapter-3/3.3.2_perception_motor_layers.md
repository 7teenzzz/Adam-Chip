# 3.3.2 Перцептивный и моторный слои

В данном разделе описывается организация перцептивного и моторного слоёв инсталляции «Адам Чип» на уровне физических компонентов. Перцептивный слой обеспечивает чувственное восприятие среды; моторный слой — физическое выражение состояний агента через «технофлору».

## Перцептивный слой

Перцептивный слой реализует сбор сенсорных данных из пространства инсталляции через два основных канала.

**Аудиоканал.** Два микрофона INMP441 в стерео-конфигурации (`audio.esp32_mic_profile: inmp441_philips32_stereo`) захватывают звук с ESP32-S3. На стороне прошивки `audioCaptureTask()` (сообщество «Mic Audio Capture (I2S)», 37 узлов в граф-отчёте ESP32) реализует непрерывную запись через шину I2S RX (BCLK=48, WS=47, SD=21) с частотой 16 кГц в формате Philips 32-bit. Захваченные данные накапливаются в кольцевом буфере 256 КБ PSRAM — примерно 4 секунды стерео-аудио — и конвертируются в PCM 16-bit (`convertRawSampleToPcm()`). Jetson получает PCM-данные через HTTP API ESP32 (`audio.mic_source: esp32`). Резервный канал — локальный PulseAudio (`audio.input_device: pulse`) — активируется при отказе ESP32-микрофона после `audio.esp_mic_fail_threshold: 6` последовательных неудач. `EspAudioHealthMonitor` непрерывно контролирует качество звука: молчание диагностируется при RMS < `esp_health.silence_threshold: 24`; клиппирование — при peak/RMS > `esp_health.ratio_threshold: 6.0`.

**Видеоканал.** Камера OV5640 захватывает MJPEG-поток, доступный по адресу `http://192.168.0.171:81/stream`. `CameraReader` (`System/adam/camera.py`, 23 рёбра в граф-отчёте) периодически получает кадры и передаёт их `SceneWorker` для VLM-анализа. При недоступности ESP32-камеры система переходит на локальный GStreamer pipeline через `/dev/video0` после `media.video.esp_fail_threshold: 3` неудач.

**Дополнительные сенсоры.** TEMT6000 фиксирует уровень освещённости окружающей среды; BTE16-19 — биометрический датчик. Данные доступны через HTTP API ESP32.

## Моторный слой: три типа технофлоры

Моторный слой представлен тремя типами «технофлоры» — физических элементов инсталляции с активным электронным поведением. Каждый тип обеспечивает отдельный сенсорный канал взаимодействия со зрителем.

**Светофлора** реализует световое поведение через 16-канальный 12-bit PWM-расширитель PCA9685. Каналы (`mcu.channels.min: 0`, `mcu.channels.max: 15`, диапазон значений 0–4095) управляют индивидуальными LED-элементами инсталляции. На стороне прошивки `Pca9685Module` (сообщество «PCA9685 PWM Control», 15 узлов) хранит частоту PWM и активную сцену в энергонезависимой памяти NVS (`nvsSaveFrequency()`, `nvsSaveScene()`) — выставочное состояние сохраняется при отключении питания и автоматически восстанавливается при перезагрузке. Сцены задают паттерны активации каналов: `boot_idle` — спокойное фоновое мерцание; `all_on` — полное освещение; `alternating` — чередующаяся активность. Через whitelist `mcu.allowed_scenes` гарантируется, что языковая модель не может активировать произвольные паттерны.

**Аудиофлора** реализует звуковое поведение через ЦАП PCM5102A с усилителем PAM8403. На стороне прошивки `speakerPlaybackTask()` (сообщество «Speaker Playback and Sounds», 16 узлов) принимает PCM-данные в формате моно 16-bit 44,1 кГц через I2S TX (BCLK=38, LRCK=39, DATA=40) и буферизует их в 32 КБ кольцевом буфере DRAM (~372 мс звука). Запись в буфер rate-limited с 4 мс задержкой I2S drain вместо отбрасывания кадров — это предотвращает аудиоглитчи при высокой нагрузке. Jetson отправляет аудиоданные на отдельный HTTP-сервер ESP32 (`mcu.speaker_url: http://192.168.0.171:81/speaker`, порт 81). Аудиофлора воспроизводит как TTS-речь с Jetson, так и системные звуки прошивки (`playSystemSound()`, 12 рёбер в граф-отчёте).

**Виброфлора** реализует тактильное поведение через прямое управление GPIO ESP32. Вибромоторы создают ощущение «дыхания» или «пульса» инсталляции. Параметры активации ограничены `safety.motor_max_duration_ms: 2500` и `safety.motor_cooldown_ms: 250`.

## Управление безопасностью

Все моторные команды проходят через `ActionLayer` до достижения `MCUClient`. Параметры безопасности заданы в разделе `safety` конфигурации и жёстко соблюдаются: превышение `motor_max_duration_ms` ведёт к автоматическому ограничению; нарушение `motor_cooldown_ms` — к очередизации команд. При потере связи с ESP32 (`mcu.timeout_sec: 1`) система переходит к сцене `mcu.idle_scene: boot_idle` — защитное поведение по умолчанию.

## Связь с критерием воплощённости (2.1.7)

Три типа технофлоры совместно формируют **мультимодальное телесное присутствие** агента: свет + звук + вибрация создают интегрированный перцептивный опыт, недостижимый ни одним каналом по отдельности. Этот синергетический эффект рассматривается в разделе 3.4.5 как один из наблюдаемых эмерджентных эффектов системы.
