# 3.4.1 Задачи и методика тестирования

В данном разделе определяются задачи тестирования нейроагентной инсталляции «Адам Чип», методика сбора данных и инфраструктура автоматического логирования. Тестирование направлено на оценку соответствия поведения агента теоретическим критериям квазисубъектности (раздел 2.1) и выявление ограничений реализованной архитектуры.

## Цели тестирования

Тестирование нейроагентной инсталляции решает три группы задач.

**Верификация (Verification)**: проверка соответствия реализованных компонентов архитектурным спецификациям — наличие всех заявленных модулей, корректность их взаимодействия, выполнение инвариантов безопасности. Данная группа задач решается через диагностические скрипты (раздел 3.3.5) и статический анализ кода.

**Оценка (Evaluation)**: измерение количественных показателей поведения агента в условиях реального взаимодействия — латентности, точности распознавания, устойчивости идентичности, качества памяти. Задачи оценки операционализируют критерии квазисубъектности через измеримые метрики.

**Апробация (Validation)**: качественная оценка соответствия художественному замыслу — воспринимается ли агент как обладающий «присутствием», согласованно ли его поведение, создаёт ли инсталляция желаемый опыт взаимодействия.

## Инфраструктура логирования

Методика тестирования опирается на встроенную инфраструктуру автоматического логирования, реализованную в системе.

`EventLog` (`System/adam/events.py`, 13 рёбер) — централизованный event bus: все модули системы публикуют события в единый поток, который сохраняется в `data/adam/events.jsonl` в формате JSONL. Каждое событие содержит: timestamp, тип события (`asr_result`, `llm_response`, `tts_start`, `action_result` и др.), `turn_id` (UUID, связывающий все события одного диалогового обмена), значение метрики или payload.

`metrics.py` (`System/adam/metrics.py`) — модуль сбора метрик латентности: фиксирует время начала и окончания каждой стадии turn'а (ASR, LLM, TTS, Action), вычисляет latency per stage и total turn latency.

`/api/agent/turns` — REST-эндпоинт агрегации: принимает limit и возвращает структурированные записи по turn_id с агрегированными метриками и списком событий каждого turn'а.

Log Viewer (`System/adam/log_viewer.py`, порт 8083) — всегда доступный read-only HTTP-сервис: позволяет просматривать события, метрики и статусы сервисов в реальном времени даже при остановленном оркестраторе.

Удалённый сбор данных для анализа на Windows/macOS: `scripts/adam_pull_logs.py --last N --out json > turns.json`.

## Методика измерений

Измерения проводятся в двух условиях: **controlled** (лабораторный тест с фиксированными сценариями) и **exhibition** (наблюдение в ходе реальных показов инсталляции).

Для controlled-тестирования разработаны тестовые сценарии: фиксированные транскрипции, отправляемые через `/api/agent/turn` API, позволяют воспроизводить идентичные условия и сравнивать результаты. Exhibition-наблюдение фиксирует реальное поведение в условиях неконтролируемого взаимодействия с посетителями.

Единица измерения — **диалоговый turn** (один цикл запрос → ответ), идентифицируемый через `turn_id`. Сессия — совокупность turn'ов между включением и выключением оркестратора.
