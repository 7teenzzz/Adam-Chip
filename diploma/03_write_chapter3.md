# Stage 3: Chapter 3 Writing — Per-Section Academic Russian

## ROLE

Ты выступаешь как научный писатель и редактор. Стиль — академический русский язык, выпускная квалификационная работа, требования ГОСТ.

## INPUT

При каждом запуске получаешь:

1. **Целевой раздел** — один из 19 разделов главы 3 (см. `diploma/chapter_map.md`)
2. **Verification-материалы** — соответствующий файл из `diploma/project-verification/`:
   - `by-section/3.X_*.md` (общая верификация раздела)
   - `by-criterion/crit_NN_*.md` (если раздел касается одного из 8 критериев)
3. **Архитектурный контекст** — `project-analysis/synthesis/master_concepts.md`
4. **Источники** — `diploma/Diploma_bibliography.md`
5. **Граф кода** — `graphify-out/graph.json` (доступен через `/graphify query`)
6. **Граф персонажа** — `graphify-out-persona/graph.json`

## OBJECTIVE

Произвести **один markdown-файл** в `diploma/chapter-3/<номер>_<slug>.md` — связный академический текст ≥1500 знаков, готовый к включению в выпускную работу.

## STYLE RULES

### Грамматика и лексика

- **Пассивный залог и безличные конструкции** преобладают: «Реализована система …», «Было разработано …»
- **Прошедшее время** для описания того, что сделано (раздел 3.1, 3.2, 3.3)
- **Настоящее время** для описания теоретических положений и общих принципов
- **Будущее время** избегать (за исключением «в дальнейшем планируется»)
- **Личных местоимений нет** (никаких «я», «мы делаем»). Допустимо: «в работе рассмотрено», «в рамках исследования»
- **Жаргона нет.** «нейроагент», а не «бот»; «языковая модель», а не «LLM» при первом упоминании (потом можно ввести сокращение)

### Структура раздела

- Каждый раздел начинается с **одного абзаца-введения** (что в нём рассматривается)
- Далее — основной текст с подразделами при необходимости (через `###`)
- В конце — **связка со следующим разделом** (1–2 предложения)
- При первом упоминании любой технологии или термина — определение + ссылка на источник

### Ссылки на источники

- Формат ГОСТ: `[N]` где `N` — номер из `Diploma_bibliography.md`
- Если источник нужен, но его нет в библиографии — добавить в конец раздела блок:
  ```markdown
  > **TODO (источник):** требуется ссылка на: <описание>
  ```
  Это пометит для пользователя, что библиографию нужно дополнить.

### Связь с архитектурой

Для разделов 3.2–3.4 обязательно ссылаться на конкретные модули кода:
- `EpisodicMemory` (System/adam/episodic.py) — корректный формат
- При описании архитектурного решения — короткий код-блок или таблица параметров из `Config.json`

### Связь с критериями

Для разделов 3.4.2–3.4.5 явно указывать, какой из 8 критериев квазисубъектности (раздел 2.1) проверяется:

> «Метрики удержания роли, рассмотренные в данном разделе, операционализируют критерий устойчивости идентичности (раздел 2.1.3) и критерий нормативности (раздел 2.1.4).»

## CONSTRAINTS

- **Не повторять содержание других разделов.** Если факт уже описан в 3.2.4, в 3.4.3 — ссылка `(см. раздел 3.2.4)`, не повтор.
- **Не выдумывать архитектурные решения, которых нет в коде.** Если verification-файл говорит `MISSING` — описать это честно как ограничение, а не fake-implementation.
- **Не пересказывать диплом.** Цель — описать **реализацию**, а не теорию. Теория уже есть в главах 1–2.
- **Не превращать в туториал.** Это не документация — это академическое описание системы.
