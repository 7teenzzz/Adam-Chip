# Диплом Adam Chip: IMRAD-подобная структура

## Проблема: IMRAD vs Диплом

**IMRAD (научная статья):**
```
I  Introduction      → why this problem?
M  Methods           → how you did it
R  Results           → what you found
A/D Analysis/Discuss → what it means
```

**Диплом (выпускная работа):**
```
Введение      → актуальность, задачи, объект/предмет
Глава 1       → теоретический фундамент (philosophy + tech)
Глава 2       → архитектурные модели (case studies, критерии)
Глава 3       → реализация (implementation + verification)
Источники     → библиография
```

## На самом деле диплом — это **IMRAD с предисловием**

| IMRAD | Диплом | Что там |
|-------|--------|---------|
| **I** Introduction | Введение + Глава 1 | "Why?" (актуальность, философия) |
| **I** Introduction | Глава 2 | "What exists?" (case studies, criteria) |
| **M** Methods | Глава 3.1–3.2 | "How?" (архитектура, стек) |
| **R** Results | Глава 3.3–3.4 | "What was built?" (инсталляция, тесты) |
| **A/D** Analysis | Глава 3.4–3.5 | "What does it mean?" (интерпретация) |

---

## Глава 3 как вложенный IMRAD

**Глава 3 структурирована как микро-IMRAD:**

```
3.1 Концепция проекта
    └─ M (Methods): "Мы выбрали такую-то архитектуру потому что..."

3.2 Разработка приложения
    └─ M (Methods): "Компоненты реализуются так..."
    └─ R (Results): "Архитектура состоит из..."

3.3 Разработка инсталляции  
    └─ M (Methods): "Инсталляция собрана из..."
    └─ R (Results): "Система работает следующим образом..."

3.4 Тестирование и апробация
    └─ M (Methods): "Методика тестирования..."
    └─ R (Results): "Получены метрики..."
    └─ A/D (Analysis): "Результаты показывают, что..."
```

## Адаптированный IMRAD для Главы 3

### I — Introduction: Связь с критериями (новый раздел 3.0)

**Что описывается (ОДИН раздел в начале Главы 3):**
- Переход от теории (главы 1–2) к реализации
- Какие из 8 критериев квазисубъектности (2.1.1–2.1.8) адресирует Adam Chip?
- Какие критерии критичны для художественной инсталляции?
- Как теоретическая архитектура из 2.4.3 переводится в инженерные требования?
- Таблица соответствия: какой компонент 3.X реализует какой критерий из 2.1

**Откуда берётся:**
- Итоги Главы 2 (синтез, выводы 2.4)
- Неявные требования из 3.1 (концепция проекта)
- Таблица из chapter_map.md (связи 3.4 ↔ 2.1)

**Стиль:** Bridging section, не повторение. Таблица, а не пересказ.

**Длина:** ~500–800 слов. **Это INTRO ко всей главе 3**, а не к отдельному разделу.

---

### M — Methods: как реализовано (3.1 + 3.2 + 3.3.1–3.3.3)

**Что описывается:**

**3.1: Концепция проекта** (БЕЗ обоснования, только описание)
- Художественный замысел (инсталляция, персонаж, взаимодействие)
- Логика поведения агента (как он должен себя вести)
- Функции в структуре инсталляции

**3.2: Разработка приложения** (чистая техническая спецификация)
- Общая архитектура (диаграмма, модули)
- Программный стек (язык, фреймворки, версии, зависимости)
- Компоненты:
  - Системный промпт и конфигурация идентичности (AIIM, Config.json)
  - Память (episodic.py, memory.py, consolidator.py)
  - Инференс (LLM/VLM/ASR/TTS адаптеры)
  - Action layer (валидация команд, MCU client)
  - Event bus и logging
- Параметры конфигурации (из Config.json и Config.schema.json)

**3.3: Разработка инсталляции** (железо + встроенная система)
- Jetson Orin NX Super (архитектура, JetPack версия, CUDA)
- ESP32-S3 WROOM CAM (компоненты: PCA9685, INMP441, OV5640, PCM5102A, W5500)
- Каналы восприятия (видео MJPEG, микрофон WebRTC VAD, текст)
- Каналы действия (TTS через HDMI, PWM для света/движения)
- Прошивка ESP32 (PlatformIO, архитектура)
- Сетевая топология и протоколы (HTTP, WebSocket, Ethernet SPI)

**Стиль:** 
- Только ФАКТЫ и ПАРАМЕТРЫ, без "почему" (почему в главе 2)
- Воспроизводимость: версии, пути к файлам, конфиги, pin-outs
- **Если кто-то прочитает этот раздел в отрыве, сможет воссоздать систему**

**Длина:** ~6000–8000 слов + диаграммы.

### R — Results: что получилось (3.3.4 + 3.3.5 + 3.4.1–3.4.4)

**Что описывается:**

**3.3.4: Сценарий взаимодействия**
- Пример диалога со зрителем (расшифровка: восприятие → интерпретация → планирование → действие)
- Примеры разных типов взаимодействия (первый раз, повторный, конфликт)
- Как система обрабатывает случаи (с примерами из логов)

**3.3.5: Тестирование и апробация**
- Методика (как тестировалось, где, на ком)
- Сценарии тестирования
- Инструменты измерения

**3.4.1–3.4.4: Метрики и результаты**
- **3.4.2: Метрики удержания роли и нормативной устойчивости** (consistency score, role retention examples)
- **3.4.3: Метрики памяти и темпоральной связности** (recall rate, consolidation latency, episodic density)
- **3.4.4: Метрики интеракционности и временной эффективности** (ASR latency, LLM latency, TTS latency, response time, initiative rate)
- **3.4.5: Интерпретация результатов и ограничения** (честная оценка: что работает, что не работает)

**Стиль:**
- ФАКТЫ: числа, таблицы, примеры, графики
- БЕЗ ИНТЕРПРЕТАЦИИ (это будет в A/D)
- Честно про ограничения и баги

**Данные:**
- Таблица: ASR latency ~1.5s, LLM latency ~9s (SWA cache), TTS latency ~0.8s
- Примеры диалогов (расшифровки из логов)
- Графики (если есть данные по времени)

**Длина:** ~3000–4000 слов + таблицы.

### A/D — Analysis/Discussion: Интерпретация (3.4.5 + новое 3.5)

**Что описывается:**

**В конце 3.4.5:**
- Что показали метрики относительно критериев из 2.1?
  - Критерий 2.1.3 (идентичность): реализован? На 100%? На 70%?
  - Критерий 2.1.5 (темпоральная связность): память работает?
  - Критерий 2.1.6 (интеракционность): агент реагирует на зрителя?
  - И т.д. для каждого из 8 критериев

**Новый раздел 3.5: Заключение главы**
- Как архитектура Adam Chip соотносится с моделями из 2.2–2.3 (case studies)?
  - Agent Ruby: что общего? что отличается?
  - Being: как сравнить?
  - Voyager, Generative Agents: какие lessons?

- Что удалось реализовать? Почему?
- Что не удалось? Почему?
- Ограничения:
  - Технические (Jetson VRAM 16GB, SWA cache 512, Jetson PyTorch на Jetson)
  - Архитектурные (отсутствие Wave 2: neural search)
  - Практические (время разработки, сложность интеграции)

- Что было неожиданным?
- Что получилось лучше планов?
- Что следовало бы делать иначе?

**Стиль:**
- АНАЛИЗ, не пересказ результатов
- Честность: противоречия, проблемы, неудачи
- Связь с теорией: возврат к главам 1–2
- Критический взгляд, а не защита

**Длина:** ~2000–2500 слов.

## Как адаптировать IMRAD для writing Главы 3

### Шаг 1: Раскрой каждый подраздел 3.X в IMRAD-структуру

**Пример 3.2.3 "Системный промпт и конфигурация идентичности":**

```
I: Почему системный промпт критичен для удержания роли? 
   → ссылка на 2.1.3 (устойчивость идентичности), 2.1.4 (нормативность)
   → ссылка на 1.2.4 (агентность и идентичность)

M: Как устроен системный промпт Adam Chip?
   → AIIM-формула (таблица из диплома 3.1.1)
   → Config.json параметры
   → граф: /graphify query "AIIM" / "persona_paths" / "Identity.md"

R: Какой результат?
   → Примеры: как промпт формирует реакции
   → Метрика: удержание роли (если измеряли)
   → Логи из memory.sqlite3

A/D: Что это говорит о квазисубъектности?
   → Критерий 2.1.3 (идентичность): FULL / PARTIAL / MISSING
   → Критерий 2.1.4 (нормативность): ...
   → Ограничения: длина контекста, версия LLM
```

### Шаг 2: Используй paper-writer скилл, но адаптируй промпт

Вместо: "write paper on medical topic"
Используй: "write diploma chapter 3 section 3.2.3 following adapted IMRAD: I (why system prompt?), M (how implemented?), R (what works?), A (what criteria does it satisfy?)"

### Шаг 3: Проверь каждый раздел на полноту

**Контрольный лист для раздела:**

- [ ] **I** есть? Обоснована ли необходимость этого компонента теорией (главы 1–2)?
- [ ] **M** есть? Описаны ли конкретные линии кода, параметры, граф?
- [ ] **R** есть? Есть ли примеры, метрики, доказательства что работает?
- [ ] **A/D** есть? Связь с какими критериями из 2.1? Какие ограничения?
- [ ] **Не повторяется ли** содержание других разделов?
- [ ] **Источники** — все ли техские детали трассируемы через `/graphify query`?

## ИТОГОВАЯ РЕКОМЕНДАЦИЯ

**Используй paper-writer скилл + adapted IMRAD + этот файл как шпаргалку.**

1. **Для каждого подраздела 3.X:**
   - Раскрой IMRAD-структуру (что выше)
   - Запусти `/paper-writer "написать раздел 3.X следуя IMRAD: I (почему?), M (как?), R (результаты?), A/D (что значит?)"`
   - Скилл создаст struktura файл
   
2. **Проверяй через graphify:**
   - `/graphify query "<ключевой концепт из раздела>"` → runtime-evidence
   - Дополняй примеры, метрики, примеры диалогов

3. **Финальная верификация:**
   - Каждый раздел → контрольный лист выше
   - Никаких повторов между разделами
   - Все ссылки на код трассируемы

Глава 3 должна получиться ~20–25 страниц, связная, с примерами и данными.
